## 一、 数据库系统 (Database Systems)

### 1. 理论学习

#### 1. 事务管理与 ACID 特性底层原理
- **原子性（Atomicity）与 Undo Log**：  
  事务是数据库操作的最小逻辑单位。原子性规定事务内的操作要么全部提交成功，要么全部回滚。
  *底层机制*：数据库引入了**回滚日志（Undo Log）**。在事务中对数据进行任何修改（`INSERT`/`UPDATE`/`DELETE`）前，系统先将该操作的反向逆操作（例如，若执行插入，日志则记录删除；若执行修改，日志则记录修改前的值）写入 Undo Log。一旦事务失败或显式调用 `ROLLBACK`，引擎读取 Undo Log 逆向执行，从而将数据恢复到事务开始前的状态。
- **持久性（Durability）与 Redo Log / WAL**：  
  持久性保障事务一旦提交，其修改就永久保存在物理磁盘中。
  *底层机制*：为提升性能，数据库修改数据是先写内存缓冲区（Buffer Pool）再异步刷盘。为防宕机丢失，采用 **WAL（Write-Ahead Logging，预写日志）** 策略：在事务提交前，先把修改记录写入物理**重做日志（Redo Log）**并强制刷盘（顺序 I/O 速度极快）。当系统崩溃重启时，通过重放 Redo Log 重新把修改应用到物理页（页重构），确保数据不丢失。
- **隔离性（Isolation）与锁/MVCC**：  
  隔离性防止并发事务交错执行时导致的数据混乱。由排他锁/共享锁以及多版本并发控制（MVCC）共同实现。
- **一致性（Consistency）**：  
  一致性是事务的最终追求，要求数据库从一个合法状态转移到另一个合法状态，需完整性约束（外键、触发器、非空限制）以及原子性、隔离性、持久性共同支撑。

#### 2. 并发隔离级别与 MVCC 机制
- **隔离级别与并发缺陷对照表**：
  | 隔离级别 | 脏读（Dirty Read） | 不可重复读（Non-repeatable Read） | 幻读（Phantom Read） |
  | :--- | :---: | :---: | :---: |
  | **读未提交（Read Uncommitted）** | 允许 | 允许 | 允许 |
  | **读已提交（Read Committed）** | 不允许 | 允许 | 允许 |
  | **可重复读（Repeatable Read）** | 不允许 | 不允许 | 允许（InnoDB 依靠 Next-Key 锁解决） |
  | **串行化（Serializable）** | 不允许 | 不允许 | 不允许 |
- **MVCC (Multi-Version Concurrency Control) 多版本并发控制机制**：  
  MVCC 允许数据库在执行“读写”并发时**不加锁**，极大提升了并发吞吐量。以 MySQL InnoDB 为例：
    - **隐藏列**：每行记录后均包含三个隐藏列：`DB_TRX_ID`（最近修改该行的事务 ID）、`DB_ROLL_PTR`（回滚指针，指向 Undo Log 中的历史版本链）和 `DB_ROW_ID`。
    - **ReadView (读视图)**：当事务发起“快照读”（普通 `SELECT`）时产生。ReadView 包含：当前活跃的未提交事务 ID 列表（`m_ids`）、最小活跃事务 ID（`min_trx_id`）和下一个将要分配的事务 ID（`max_trx_id`）。
    - **版本链可见性规则**：读取时沿着 Undo Log 版本链回溯，若某版本的 `DB_TRX_ID` 小于 `min_trx_id`（说明该版本在 ReadView 创建前已提交），则该版本可见；若大于 `max_trx_id`（在创建后才开启的事务），则不可见；若在两者之间，则判断其是否在 `m_ids` 列表中（未提交则不可见，已提交则可见）。

#### 3. 索引体系与查询优化
- **B+ 树索引结构**：  
  B+ 树的叶子节点存放全部键值和数据，非叶子节点只作为目录索引。这保证了查找任意键值的磁盘 I/O 次数相同且极少。
- **聚簇索引（Clustered Index）与二级索引（Secondary Index）**：  
  聚簇索引的叶子节点直接存放行数据（数据与索引合一，每张表仅能有一个）；二级索引（非聚簇索引，如联合索引、单列辅助索引）叶子节点存放的是主键值。访问二级索引需要先查到主键，再到聚簇索引中“**回表**”查询行数据。

### 2. 课后习题

#### Q1. 数据库事务的四大特性（ACID）中，用来保障“事务对数据的修改在系统崩溃重启后依然完好”的特性及其底层组件是：
- A. 原子性，由 Undo Log 保障
- B. 一致性，由 Buffer Pool 保障
- C. 持久性，由 Redo Log 与 WAL 机制保障
- D. 隔离性，由 MVCC 保障
- **正确答案**：C
- **解析**：持久性保障事务提交后永久有效。InnoDB 引擎依靠 Redo Log 顺序预写机制（WAL），在系统意外宕机重启后通过重做 Redo Log 恢复未刷盘的脏页，确保数据持久化。

#### Q2. 并发事务中，事务 A 读取了事务 B 已经修改但尚未提交的数据，之后事务 B 发生回滚。这种并发缺陷称为：
- A. 脏读（Dirty Read）
- B. 不可重复读（Non-repeatable Read）
- C. 幻读（Phantom Read）
- D. 丢失修改（Lost Update）
- **正确答案**：A
- **解析**：脏读是指读取到了并发事务未提交的数据，如果该事务随后回滚，则读取的数据就是无效的“脏”数据。

#### Q3. MySQL InnoDB 引擎默认的事务隔离级别是：
- A. 读未提交（Read Uncommitted）
- B. 读已提交（Read Committed）
- C. 可重复读（Repeatable Read）
- D. 串行化（Serializable）
- **正确答案**：C
- **解析**：MySQL 默认隔离级别为 Repeatable Read。在此级别下，同一事务内多次读取同一行数据的结果是恒定一致的。InnoDB 在此级别下结合 MVCC 和 Next-Key Locks（临键锁）基本解决了幻读问题。

#### Q4. 关于 MVCC（多版本并发控制）的运作机制，下列说法错误的是：
- A. MVCC 使得对同一行数据的读操作和写操作可以并行进行，无需相互等待锁释放
- B. 它通过在物理磁盘中复制几十个相同的物理数据库文件来实现版本控制
- C. 它主要依赖记录中的隐藏列、Undo Log 版本链以及 ReadView 来判断版本的可见性
- D. 在读已提交（RC）隔离级别下，每次普通 `SELECT` 都会生成一个新的 ReadView
- **正确答案**：B
- **解析**：MVCC 并不是复制物理数据库文件，而是利用 Undo Log 机制将每次修改前的旧状态以链表形式连接，配合只读的 ReadView 快照判定逻辑来实现多版本共存。

#### Q5. 在二级索引（非聚簇索引）中，如果需要查询非索引字段的数据，通常会发生的动作为：
- A. 锁表
- B. 重新生成聚簇索引
- C. 回表查询（先查到主键，再用主键在聚簇索引中定位行数据）
- D. 数据库崩溃
- **正确答案**：C
- **解析**：二级索引叶子节点只存储了主键 ID。如果要查询非该索引包含的字段，必须拿着主键 ID 去聚簇索引树上再次查找整行数据，这一过程称为“回表”。

#### Q6. 设有一个联合索引 `idx_a_b_c (a, b, c)`，下列 SQL 语句中，无法使用该索引（完全失效）的是：
- A. `SELECT * FROM tbl WHERE a = 1;`
- B. `SELECT * FROM tbl WHERE a = 1 AND b = 2;`
- C. `SELECT * FROM tbl WHERE b = 2 AND c = 3;`
- D. `SELECT * FROM tbl WHERE a = 1 AND c = 3;`
- **正确答案**：C
- **解析**：根据**最左前缀匹配原则**，查询条件必须包含联合索引的最左侧字段（此处为 `a`），索引才可能生效。C 选项的条件只包含了 `b` 和 `c`，缺失了 `a`，导致索引完全失效，触发全表扫描。D 选项能用到 `a` 字段的索引过滤。

#### Q7. 数据库中“幻读”（Phantom Read）缺陷的具体表现是：
- A. 同一事务中多次读取同一行，发现字段数值被他人修改了
- B. 读取到了他人已经修改但尚未提交的数据
- C. 同一事务中根据相同条件进行范围读取，发现多出了（或减少了）他人新插入（或删除）的行
- D. 数据库发生死锁，无法输出结果
- **正确答案**：C
- **解析**：幻读侧重于**范围查询中行数的变化**（如原本查出 5 行，第二次查出 6 行，多出了一行像幻觉一样）。A 选项是不可重复读，描述的是同一行数据内容被改；B 选项是脏读。

#### Q8. MySQL InnoDB 依靠什么锁机制在“可重复读”隔离级别下防止幻读：
- A. 行锁（Record Lock）
- B. 表锁（Table Lock）
- C. 间隙锁（Gap Lock）与临键锁（Next-Key Lock）
- D. 共享锁（S 锁）
- **正确答案**：C
- **解析**：临键锁（Next-Key Lock）是行锁和间隙锁的结合，它不仅锁住记录行，还会锁住记录行之间的间隙，防止其他并发事务在这个范围内插入新记录，从而在 RR 级别下防御了幻读。

#### Q9. 数据库设计中，若将一个关系模式分解为满足“消除了非主属性对码的部分函数依赖”的要求，该模式达到了：
- A. 第一范式（1NF）
- B. 第二范式（2NF）
- C. 第三范式（3NF）
- D. BCNF 范式
- **正确答案**：B
- **解析**：
  - 1NF 要求字段原子性。
  - 2NF 在满足 1NF 基础上，消除非主属性对码的部分依赖。
  - 3NF 消除传递依赖。

#### Q10. 在 SQL 优化中，使用 `EXPLAIN` 关键字查看执行计划，如果 `type` 字段显示为 `ALL`，意味着：
- A. 使用了全索引扫描，效率极高
- B. 执行了全表扫描（Full Table Scan），效率最低，通常需要建立索引优化
- C. 发生死锁，执行终止
- D. 查询条件语法错误
- **正确答案**：B
- **解析**：在 `EXPLAIN` 结果中，`type` 代表访问类型。`ALL` 代表全表扫描，意味着数据库需要从头到尾读取整张表的文件，对于大表而言性能开销极大，必须予以索引优化。

## 二、 计算机网络 (Computer Networks)

### 1. 理论学习

#### 1. 体系结构与 TCP 可靠传输机制
- **OSI 七层模型与 TCP/IP 四层架构**：
    - OSI：物理层、数据链路层、网络层（IP）、传输层（TCP/UDP）、会话层、表示层、应用层。
    - TCP/IP：网络接口层、网络层、传输层、应用层。
- **TCP 三次握手过程与状态转移**：  
  三次握手用于同步双向的初始序列号（ISN）并确认双方的接收与发送能力。
    1. **第一次握手**：客户端发送连接请求报文（SYN = 1, seq = x），客户端进入 `SYN_SENT` 状态。
    2. **第二次握手**：服务端收到后，同意连接则返回响应报文（SYN = 1, ACK = 1, seq = y, ack = x + 1），服务端进入 `SYN_RCVD` 状态。
    3. **第三次握手**：客户端收到响应，返回确认报文（ACK = 1, seq = x + 1, ack = y + 1），客户端进入 `ESTABLISHED` 状态。服务端收到后也进入 `ESTABLISHED`。
- **TCP 四次挥手与 TIME_WAIT 状态**：  
  由于 TCP 是全双工的，每个方向的连接关闭都需要单独发送 FIN 包和接收 ACK 确认。主动关闭方在收到被动方的最后一个 FIN 并发送 ACK 后，必须进入 **`TIME_WAIT` 状态并等待 2MSL（最大报文寿命两倍）** 的时间。
  *TIME_WAIT 的两个核心作用*：
    1. **保证可靠关闭**：防止最后一个 ACK 丢失。如果丢失，被动方会重发 FIN，主动方在 2MSL 内可以重新发送 ACK。
    2. **清理残留报文**：使当前连接产生的所有网络报文在网络中过期消失，防止这些旧报文在新建立的相同四元组连接中被误收造成干扰。

#### 2. TCP 拥塞控制机制
- **控制算法演进**：  
  拥塞控制用于防止过多数据涌入网络造成网络过载。发送端维持一个**拥塞窗口（cwnd）**，发送速率受限于 `min(cwnd, rwnd)`（接收窗口）。
    - **慢启动（Slow Start）**：初始 `cwnd = 1`。每收到一个 ACK，`cwnd` 翻倍（指数级增长），直到达到慢启动阈值 `ssthresh`。
    - **拥塞避免（Congestion Avoidance）**：达到 `ssthresh` 后，`cwnd` 每个往返时间（RTT）只增加 1（线性增长）。若发生网络拥塞（超时），则 `ssthresh` 降为当前 `cwnd` 的一半，`cwnd` 重置为 1，重新进入慢启动。
    - **快重传（Fast Retransmit）**：接收端收到失序包时立即发送重复 ACK。发送端连续收到 3 个重复 ACK 时，认定该包丢失，立即重传，无需等待超时。
    - **快恢复（Fast Recovery）**：配合快重传。此时不将 `cwnd` 降为 1，而是降为原先的一半，`ssthresh` 也设为该值，然后直接开始拥塞避免算法（线性增长）。

#### 3. 应用层协议：HTTP 与 HTTPS 安全通道
- **HTTP 1.0、1.1、2.0 与 3.0 的区别**：
    - *HTTP 1.1*：引入长连接（Keep-Alive）、管道化技术（Pipeline）。
    - *HTTP 2.0*：多路复用（Multiplexing，单 TCP 连接上并发双向流，解决队头阻塞）、二进制分帧、头部压缩（HPACK）、服务器推送。
    - *HTTP 3.0 (QUIC)*：弃用 TCP，基于 **UDP** 协议实现，彻底解决了 TCP 的传输层队头阻塞问题，支持极速 0-RTT 连接建立。
- **HTTPS 与 TLS 安全握手机制**：  
  HTTPS = HTTP + SSL/TLS。TLS 握手利用了**非对称加密**与**对称加密**的结合：
    1. 客户端向服务端发起请求，服务端返回经过权威 CA 机构签名的**数字证书**（内含服务端的公钥）。
    2. 客户端校验证书真伪。若合法，在本地生成一个随机的对称密钥（Pre-Master Secret）。
    3. 客户端用证书里的**公钥**对该对称密钥加密，发送给服务端。
    4. 服务端用自己的私钥解密，获得对称密钥。
    5. 之后的所有应用层通信均使用该**对称密钥**进行加密传输（对称加密速度快，适合大数据量传输）。

### 2. 课后习题

#### Q1. TCP 建立连接的三次握手过程中，客户端收到服务端的 SYN + ACK 报文后，处于的状态是：
- A. `SYN_SENT`
- B. `SYN_RCVD`
- C. `ESTABLISHED`
- D. `TIME_WAIT`
- **正确答案**：C
- **解析**：客户端收到第二步的 SYN + ACK 报文后，向服务端回复最后的 ACK，此时客户端便完成了它这一端的连接建立，状态变迁为 `ESTABLISHED`。服务端收到最后的 ACK 后也转为 `ESTABLISHED`。

#### Q2. 在 TCP 断开连接的四次挥手协议中，主动关闭方进入 `TIME_WAIT` 状态并必须等待 2MSL 时间，其主要目的是：
- A. 确保被动方发送的数据全部被接收完毕
- B. 保证最后一个确认 ACK 报文能到达被动方以安全关闭连接，并使网络中所有残留报文过期失效
- C. 提高 TCP 的数据吞吐量
- D. 重新计算拥塞控制窗口大小
- **正确答案**：B
- **解析**：2MSL 的等待时间可确保因网络延迟的旧报文自然消亡，并确保被动方若未收到最后的 ACK 而重发 FIN 时，主动方仍在 TIME_WAIT 状态内可对其重新确认。

#### Q3. TCP 拥塞控制算法中，当拥塞窗口（cwnd）达到慢启动阈值（ssthresh）后，拥塞窗口的增长模式转化为：
- A. 指数增长（慢启动）
- B. 线性增长（拥塞避免）
- C. 停止增长
- D. 随机变动
- **正确答案**：B
- **解析**：当 `cwnd < ssthresh` 时执行慢启动（指数翻倍）；一旦 `cwnd >= ssthresh`，为防网络过载，转为执行拥塞避免（线性每次 +1），使窗口平缓上升。

#### Q4. TCP 拥塞控制中的“快重传（Fast Retransmit）”机制，要求发送方触发立即重传的条件是：
- A. 连续收到 3 个针对同一报文的重复 ACK 确认
- B. 经历了一个完整的往返时间（RTT）仍未收到 ACK
- C. 接收端直接发送 RESET 复位包
- D. 拥塞窗口被置为 1
- **正确答案**：A
- **解析**：快重传规定，一旦发送方连续收到 3 个对同一个丢失数据包的重复确认 ACK，就表明该包已确凿丢失，立即在定时器超时前进行重传，从而提高传输效率。

#### Q5. HTTPS 在 TLS 握手过程中，用于加密后续“实际网页内容数据（应用层数据）”的密钥是：
- A. 服务端证书里的非对称公钥
- B. 客户端本地保留的非对称私钥
- C. 双方通过非对称加密协商产生的对称密钥
- D. CA 机构分发的全局公钥
- **正确答案**：C
- **解析**：非对称加密（公钥/私钥）由于计算量庞大，仅用于在 TLS 握手阶段安全地传递和协商“对称密钥”。一旦双方安全获得该“对称密钥”（Master Secret），后续应用层通信全部使用该对称密钥进行高效率加解密。

#### Q6. 在计算机网络体系结构中，IP 协议和 TCP 协议分别工作在 OSI 七层模型中的：
- A. 物理层、数据链路层
- B. 网络层、传输层
- C. 传输层、应用层
- D. 网络层、会话层
- **正确答案**：B
- **解析**：IP 协议负责主机到主机的路由寻址，属于网络层（Network Layer）；TCP 协议负责提供进程到进程的可靠端到端传输，属于传输层（Transport Layer）。

#### Q7. HTTP 2.0 协议引入的多路复用（Multiplexing）技术，主要是为了解决 HTTP 1.1 的什么问题：
- A. 单次请求数据量过大导致的网络中断
- B. 浏览器端并发 TCP 连接数受限以及应用层队头阻塞（Head-of-line Blocking）
- C. 无法加密传输内容
- D. 服务器无法主动向客户端推送数据
- **正确答案**：B
- **解析**：HTTP 1.1 中即使有 Keep-Alive，在单个 TCP 连接上也必须等前一个请求响应返回才能发下一个，容易发生队头阻塞。HTTP 2.0 在单一 TCP 连接上划分成多个独立的双向流（二进制分帧），允许请求与响应并发交错传输，彻底消除应用层队头阻塞。

#### Q8. DNS（域名系统）协议在进行域名解析时，默认使用的传输层协议是：
- A. TCP
- B. UDP
- C. IP
- D. ICMP
- **正确答案**：B
- **解析**：域名的普通查询通常只需要一次请求与一次应答，且对实时性要求高，因此 DNS 默认使用无连接、速度快的 UDP 协议（端口号 53）。只有在区域传送（Zone Transfer）或报文超长时才使用 TCP。

#### Q9. 网络层中，ARP 协议的核心作用是：
- A. 将逻辑 IP 地址转换为物理 MAC 地址
- B. 在不同网络之间进行路由选择
- C. 自动分配动态 IP 地址给主机
- D. 将域名解析为 IP 地址
- **正确答案**：A
- **解析**：ARP（Address Resolution Protocol，地址解析协议）工作在局域网内，通过广播形式寻找特定 IP 地址主机的物理网卡 MAC 地址，以完成数据帧的封装。

#### Q10. 在 TCP 首部中，用于实现滑动窗口流量控制（Flow Control）的字段是：
- A. 校验和（Checksum）
- B. 窗口大小（Window Size / rwnd）
- C. 序列号（Sequence Number）
- D. 确认号（Acknowledgment Number）
- **正确答案**：B
- **解析**：TCP 首部中的“窗口”（Window）字段是由接收方告知发送方的接收窗口大小（rwnd）。它代表当前接收缓冲区剩余可用字节数，用以动态控制发送方的发送速度，实现流量控制防止接收端溢出。

## 三、 全方位系统 (Full-Scale/Distributed Systems)

### 1. 理论学习

#### 1. 分布式系统设计理论基石
- **CAP 定理（Brewer's CAP Theorem）**：  
  一个分布式计算系统不可能同时满足以下三点：
    - **C (Consistency, 强一致性)**：所有节点在同一时刻访问的数据副本都是最新的且完全一致。
    - **A (Availability, 高可用性)**：系统在面临部分节点故障时，每个非故障节点对读写请求必须给出非错的响应（不保证是最新的）。
    - **P (Partition Tolerance, 分区容错性)**：在遇到由于网络断开导致的网络分区（Partition）时，系统依然能够提供服务。
  *CP 与 AP 的权衡*：在分布式网络中，网络分区故障（P）是必然会发生的客观事实。因此，系统设计必须在 **CP（舍弃可用性，保证强一致）** 或 **AP（舍弃强一致，保证高可用与最终一致）** 之间做出取舍。例如，金融交易系统倾向选择 CP；电商商品展示系统通常选择 AP。
- **BASE 理论（AP 的延伸）**：  
  - **Basically Available (基本可用)**：系统在出现故障时，允许部分损失（如响应时间延长、降级页面展示）。
  - **Soft State (软状态)**：允许系统内数据存在中间状态，且该状态在不影响整体可用性的前提下可以异步延迟同步。
  - **Eventually Consistent (最终一致性)**：经过一段时间的同步后，系统内所有节点的数据最终达到一致。

#### 2. 高并发缓存系统架构与防灾设计
- **缓存三大灾难性隐患及防御方案**：
    - **缓存穿透（Cache Penetration）**：  
      *成因*：查询一个**数据库和缓存中都不存在**的数据（如黑客恶意构造的非法 ID）。由于缓存不命中，每次请求都会穿透到数据库，造成数据库崩溃。  
      *防御*：1. 使用**布隆过滤器（Bloom Filter）**在缓存层前进行数据存在性预筛；2. 将空结果（如 `null`）也写入缓存并设置较短的过期时间。
    - **缓存击穿（Cache Breakdown）**：  
      *成因*：一个**热点 Key**（如爆款商品的抢购数据）在承载高并发访问的同时**突然过期**。导致瞬间海量请求全部涌入数据库，冲垮数据库。  
      *防御*：1. 读数据库重建缓存时加互斥锁（Mutex Lock），仅允许一个线程去查库写缓存，其他线程等待；2. 设置热点数据“物理上永不过期”，在后台开启异步线程定期刷新过期时间。
    - **缓存雪崩（Cache Avalanche）**：  
      *成因*：**大量缓存 Key 在同一时间大面积失效**（或缓存服务器宕机），导致原本应访问缓存的请求瞬间全部压向数据库，引发雪崩式崩溃。  
      *防御*：1. 缓存失效时间加上一个随机盐值（Random Salt），避免集体过期；2. 搭建缓存高可用集群（如 Redis Sentinel 或 Cluster）；3. 引入多级缓存。

#### 3. 分布式负载均衡与一致性哈希
- **传统哈希取模负载的痛点**：  
  如果使用 `hash(key) % N` 调度，一旦服务器节点数 $N$ 增加或减少（动态扩容/缩容/宕机），几乎所有 key 的哈希映射位置都会发生改变。这会导致分布式缓存系统中的大面积缓存失效，引发雪崩。
- **一致性哈希（Consistent Hashing）环算法**：  
  1. 将哈希值空间组织成一个首尾相连的虚拟圆环（通常大小为 $0 \sim 2^{32} - 1$）。
  2. 计算服务器节点的哈希值，将其映射到环上的特定位置。
  3. 计算数据 Key 的哈希值，同样映射到环上。
  4. **从数据位置出发，沿环顺时针寻找碰到的第一台服务器**，作为该数据的存储/服务节点。
  *扩容与缩容的优势*：当新增加一台机器时，仅影响新节点与环上逆时针方向前驱节点之间的数据，其余所有节点的映射保持不变。
  *虚拟节点（Virtual Nodes）技术*：为防止服务器节点在环上分布不均导致“数据倾斜”与负载不均，一致性哈希为每台实体机器虚拟出多个节点（如 `Node-A#1`, `Node-A#2`）均匀散落在环上，极大地平衡了负载。

### 2. 课后习题

#### Q1. 分布式系统的 CAP 定理指出，在面临必然发生的数据网络分区（P）时，设计者必须在 CP 或 AP 中抉择。选择 CP 模型意味着：
- A. 舍弃分区容错性，保证强一致和高可用
- B. 在遭遇网络断开时，为了保证数据绝对一致（C），系统会暂停或拒绝非最新数据节点的访问，暂时牺牲可用性（A）
- C. 彻底放弃数据一致性，只追求最高的速度
- D. 只能使用单机数据库运行
- **正确答案**：B
- **解析**：CP 模型（Consistency + Partition Tolerance）将一致性放在首位。如果网络断开导致两个节点无法通信，为了防止用户读到旧数据或写入不一致，系统会拒绝其中一侧分区的读写请求，即牺牲了可用性（A）。

#### Q2. 分布式 BASE 理论是针对 AP 系统的延伸，其中“BASE”是如下三个短语的缩写：
- A. 基本可用（Basically Available）、软状态（Soft State）、最终一致性（Eventually Consistent）
- B. 银行家算法（Bankers）、原子性（Atomicity）、状态机（State）、有序性（Ensure）
- C. 备份（Backup）、自适应（Adaptive）、安全性（Safety）、易用性（Easy）
- D. 负载均衡（Balance）、安全控制（Security）、评估（Evaluate）、同步（Sync）
- **正确答案**：A
- **解析**：BASE 理论是相对于 ACID 强一致要求的妥协。它提倡：基本可用（允许部分服务降级）、软状态（数据有中间过程状态）、最终一致性（不追求即时一致，但数据最终达成一致）。

#### Q3. 缓存设计中，由于黑客构造恶意非法的 ID 发起大并发查询，导致缓存和数据库中都查无此数据，请求直接拖垮数据库的现象称为：
- A. 缓存击穿
- B. 缓存雪崩
- C. 缓存穿透
- D. 缓存雪崩
- **正确答案**：C
- **解析**：缓存穿透指查询一个“不存在”的数据，缓存不命中，每次都要穿透到数据库。击穿是单个热点 Key 过期；雪崩是大量 Key 过期或缓存服务挂掉。

#### Q4. 防御高并发下“缓存穿透”隐患，工业界最常采用的高效过滤机制是：
- A. 在数据库建立主键索引
- B. 引入布隆过滤器（Bloom Filter）在缓存前预筛
- C. 强行锁定整张用户表
- D. 降低 Redis 的内存阈值
- **正确答案**：B
- **解析**：布隆过滤器空间和时间效率极高。它利用多个哈希函数对数据映射位图，能够 100% 判定“一个元素绝对不存在于系统中”。把布隆过滤器放在缓存前，可以在不读缓存不读库的情况下拦截非法请求，防范缓存穿透。

#### Q5. 热点 Key 在极高并发请求的瞬间突然过期，导致所有线程同时查库重建缓存，瞬间冲垮数据库。这称为：
- A. 缓存穿透
- B. 缓存雪崩
- C. 缓存击穿
- D. 缓存冗余
- **正确答案**：C
- **解析**：热点数据过期导致请求瞬间直达数据库的现象叫“缓存击穿”（Breakdown）。解决方案通常是在重建缓存时加互斥锁，保证仅一个请求去查库重建，其余挂起等待。

#### Q6. 为了防止大量缓存 Key 在同一时刻大面积过期而引发“缓存雪崩”，最有效的设置策略是：
- A. 将所有 Key 的生存时间（TTL）全部设置为永久有效
- B. 将各个缓存 Key 的过期时间设置加上随机扰动值（随机盐值），使其错开过期
- C. 禁止数据库执行写入操作
- D. 使用单机 Redis 运行
- **正确答案**：B
- **解析**：缓存雪崩的核心成因是“集体过期”。给每个 Key 的过期时间加上一个随机抖动（如 $5\text{分钟} \pm \text{随机数}$），能使各 Key 的过期节点均匀散落，避免峰值瞬时涌入数据库。

#### Q7. 传统哈希取模（`hash(key) % N`）负载均衡机制在节点扩容（$N$ 变化）时的致命缺陷是：
- A. 负载分配绝对不均匀
- B. 计算哈希值太慢，耗尽 CPU
- C. 几乎所有已有 Key 的映射位置全部改变，导致分布式缓存大面积失效
- D. 无法在 Linux 下运行
- **正确答案**：C
- **解析**：当节点数 $N$ 改变，原本的取模余数计算结果会大面积改变，导致几乎所有 key 原先对应的存储节点发生偏离。这在分布式缓存中意味着全网缓存不命中，会给后台数据库带来灾难。

#### Q8. 一致性哈希（Consistent Hashing）算法中，数据 Key 被定位到具体服务节点的规则是：
- A. 数据 Key 取模后存放在固定索引的服务器上
- B. 数据 Key 映射到环上，顺时针方向寻找距离最近的服务器节点
- C. 随机分发给环上的空闲服务器
- D. 发给哈希值最小的服务器
- **正确答案**：B
- **解析**：一致性哈希中，节点和 Key 均通过 Hash 映射在同一个 $0 \sim 2^{32}-1$ 的环上。定位时，Key 沿环顺时针查找，碰到的第一台服务器就是其宿主。

#### Q9. 一致性哈希为了解决因服务器节点太少而导致数据倾斜、负载不均的问题，引入的技术是：
- A. 物理扩容
- B. 虚拟节点（Virtual Nodes）
- C. 多重哈希校验
- D. 双向环链表
- **正确答案**：B
- **解析**：当物理机器较少时，它们在环上的落点容易挤在一起造成负载严重倾斜。通过为每台物理机生成几十甚至几百个虚拟副本（虚拟节点）均匀散落在环上，数据沿环寻找时就能获得极好的负载均衡度。

#### Q10. 在微服务架构中，熔断器（Circuit Breaker）的“半开状态（Half-Open）”的具体表现是：
- A. 彻底断开服务，不再接受任何请求
- B. 允许部分流量通过进行服务健康状况尝试性检测，若成功则恢复关闭状态，若失败则退回打开（熔断）状态
- C. 发生死循环，系统瘫痪
- D. 将全部流量转发到备用数据库
- **正确答案**：B
- **解析**：熔断器有三个状态：Closed（正常关闭）、Open（打开熔断，请求直接返回降级）。当熔断过一段时间后，进入 Half-Open（半开状态），允许微量测试流量去访问目标服务，如果成功则闭合恢复，如果依旧失败则重新打开熔断器保护系统。
